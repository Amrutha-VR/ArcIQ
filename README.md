## Overview

JARVIS Quiz is a full-stack AI-powered quiz application that dynamically generates multiple-choice questions on any topic using Meta Llama 3.1 8B via Hugging Face's Inference Router. Built with Next.js 14 App Router, TypeScript, Supabase for authentication and persistence, and Zustand for state management — wrapped in an Iron Man HUD aesthetic.

The core idea is that instead of static question banks, every quiz is freshly generated by a large language model based on the user's chosen topic, difficulty, and question count. This creates genuinely infinite variety across any knowledge domain.

## Architecture Decisions

### 1. API Routes as Proxy

The Hugging Face API key is never exposed to the browser. All AI calls go through `/api/generate` and `/api/chat`, which call HuggingFace internally. This is a security requirement for any production application.

```
Browser → /api/generate → HuggingFace API → parse → validate → return questions
```

### 2. Zustand over Redux or Context

Zustand was chosen for state management because:
- No boilerplate (no actions, reducers, providers)
- Built-in `persist` middleware handles localStorage sync automatically
- `partialize` option lets you control exactly what gets persisted
- Scales from simple to complex without restructuring

### 3. Supabase for Persistence

Quiz results are saved to Supabase `quiz_attempts` so they survive across sessions and devices. When a user logs in, `AuthProvider` calls `loadResults(userId)` which fetches all their attempts. Logout clears local state. This means history is always accurate regardless of which device the user is on.

### 4. Per-user Data Isolation

Row Level Security (RLS) on the `quiz_attempts` table ensures users can only read and write their own records — enforced at the database level, not just the application level.

### 5. correctAnswer Type Safety

A critical bug was discovered and fixed: the AI model occasionally returns `correctAnswer` as a string `"2"` instead of a number `2`. In TypeScript, `selectedOption === "2"` (where selectedOption is a number) is always `false`, causing correct answers to appear wrong.

**Fix applied at three levels:**
1. `safeParseInt()` in `parseAIResponse()` — normalises at parse time
2. `correctIdx = Number(question.correctAnswer)` — coerced at render time
3. `isCorrect = selectedOption === correctIdx` — pure number comparison

### 6. No resetQuiz() on Finish

`resetQuiz()` was originally called in `handleFinishQuiz()` before navigation. This wiped the Zustand store before the results page could read from it, causing blank results screens. The fix: only call `resetQuiz()` when the user explicitly navigates away from results.

### 7. Temperature 0.1 for AI

Lower temperature = more deterministic, more factual output. The default of 0.7 was causing the model to hallucinate incorrect answers. Setting temperature to `0.1` was the single most impactful change for answer accuracy.

## AI Service Integration

### Model

**Meta Llama 3.1 8B Instruct** via `https://router.huggingface.co/v1/chat/completions`

This endpoint uses the OpenAI-compatible chat completions format (role/content messages), which replaced the legacy `api-inference.huggingface.co` endpoint in 2024.

### Prompt Engineering Strategy

The main challenge with LLM-based quiz generation is:
1. Getting valid JSON output consistently
2. Ensuring `correctAnswer` index matches the actual correct option
3. Preventing factual hallucination

**Techniques applied:**

```
Temperature: 0.1          → Maximises determinism, minimises hallucination
System message            → Primes model as "factually accurate quiz generator"
Worked example in prompt  → Shows exactly how correctAnswer index works
Strict JSON validation    → validateQuestion() checks every field
safeParseInt()            → Handles string "2" vs number 2 from AI
Retry logic               → 3 attempts with 20s backoff on MODEL_LOADING
```

**Prompt structure:**
```
[SYSTEM] You are a factually accurate quiz generator. Output only raw JSON arrays.
         correctAnswer is always an integer 0-3 representing the index of the correct option.

[USER]   Generate N questions about "{topic}" at {difficulty} difficulty.
         RULES: correctAnswer is an index. options[correctAnswer] must be factually correct.
         EXAMPLE: "What is 2+2?" → options["3","4","5","6"], correctAnswer: 1 (options[1]="4")
         Return only the JSON array.
```

### JARVIS Chat Assistant

The assistant uses the same Llama 3.1 model with a quiz-context-aware system prompt. When active during a quiz, it receives the topic and difficulty and is instructed to give Socratic hints rather than direct answers.

## Setup Instructions

### Prerequisites

- Node.js 20 or higher (`node -v` to check)
- npm 10 or higher
- A [Supabase](https://supabase.com) account (free)
- A [Hugging Face](https://huggingface.co) account (free)

### Step 1 — Clone the repository

```bash
git clone https://github.com/YOUR_USERNAME/jarvis-quiz.git
cd jarvis-quiz
```

### Step 2 — Install dependencies

```bash
npm install
```

### Step 3 — Create a Supabase project

1. Go to [supabase.com](https://supabase.com) and sign in
2. Click **New Project**
3. Name it `jarvis-quiz`, set a strong DB password, choose a region near you
4. Wait ~2 minutes for the project to provision

### Step 4 — Run the database schema

In your Supabase project, go to **SQL Editor** and run:

```sql
-- User profiles
create table public.profiles (
  id uuid references auth.users on delete cascade primary key,
  username text unique,
  avatar_url text,
  total_quizzes integer default 0,
  streak integer default 0,
  last_quiz_date date,
  created_at timestamp with time zone default timezone('utc', now())
);

alter table public.profiles enable row level security;
create policy "Users can view own profile" on public.profiles for select using (auth.uid() = id);
create policy "Users can update own profile" on public.profiles for update using (auth.uid() = id);

-- Quiz attempts (persistent history)
create table public.quiz_attempts (
  id text primary key,
  user_id uuid references auth.users on delete cascade not null,
  quiz_id text not null,
  topic text not null,
  difficulty text not null,
  num_questions integer not null,
  questions jsonb not null,
  answers jsonb not null,
  score integer not null,
  percentage numeric not null,
  time_taken integer not null,
  completed_at timestamp with time zone not null,
  created_at timestamp with time zone default timezone('utc', now())
);

alter table public.quiz_attempts enable row level security;
create policy "Users can insert own attempts" on public.quiz_attempts for insert with check (auth.uid() = user_id);
create policy "Users can view own attempts" on public.quiz_attempts for select using (auth.uid() = user_id);
create policy "Users can delete own attempts" on public.quiz_attempts for delete using (auth.uid() = user_id);

-- Auto-create profile on signup
create or replace function public.handle_new_user()
returns trigger as $$
begin
  insert into public.profiles (id, username, avatar_url)
  values (new.id, new.raw_user_meta_data->>'full_name', new.raw_user_meta_data->>'avatar_url');
  return new;
end;
$$ language plpgsql security definer;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute procedure public.handle_new_user();
```

### Step 5 — Get your Supabase API keys

Go to **Settings → API** in your Supabase project:
- Copy **Project URL**
- Copy **anon public** key
- Copy **service_role** key (keep this secret)

### Step 6 — Get your Hugging Face API key

1. Go to [huggingface.co/settings/tokens](https://huggingface.co/settings/tokens)
2. Click **New token**
3. Name: `jarvis-quiz`, Role: **Read**
4. Copy the token

### Step 7 — Create environment variables

Create `.env.local` in the project root:

```env
HUGGINGFACE_API_KEY=hf_xxxxxxxxxxxxxxxxxxxx

NEXT_PUBLIC_SUPABASE_URL=https://xxxxxxxxxxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
SUPABASE_SERVICE_ROLE_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

NEXT_PUBLIC_APP_URL=http://localhost:3000
```

### Step 8 — Configure Supabase Auth redirect URLs

Go to **Supabase Dashboard → Authentication → URL Configuration** and add to **Redirect URLs**:

```
http://localhost:3000/auth/callback
```

### Step 9 — (Optional) Set up Google OAuth

1. Go to [console.cloud.google.com](https://console.cloud.google.com)
2. Create a project → **APIs & Services → Credentials**
3. Create **OAuth 2.0 Client ID** → Web application
4. Add to **Authorized redirect URIs**:
   ```
   https://YOUR_PROJECT_REF.supabase.co/auth/v1/callback
   ```
5. Copy **Client ID** and **Client Secret**
6. In Supabase → **Authentication → Providers → Google** → enable → paste both keys

### Step 10 — Run the development server

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000).

### Step 11 — Build for production

```bash
npm run build
npm start
```

## Database Schema

```
auth.users (Supabase managed)
    ↓
public.profiles
  - id (uuid, FK → auth.users)
  - username (text)
  - avatar_url (text)
  - total_quizzes (integer)
  - streak (integer)
  - last_quiz_date (date)
  - created_at (timestamptz)

public.quiz_attempts
  - id (text, PK — UUID from client)
  - user_id (uuid, FK → auth.users)
  - quiz_id (text)
  - topic (text)
  - difficulty (text)
  - num_questions (integer)
  - questions (jsonb — full Question[] array)
  - answers (jsonb — full UserAnswer[] array)
  - score (integer)
  - percentage (numeric)
  - time_taken (integer, seconds)
  - completed_at (timestamptz)
  - created_at (timestamptz)
```

All tables have **Row Level Security** enabled. Users can only access their own rows.

## Known Limitations

| Limitation | Impact | Potential Fix |
|---|---|---|
| Hugging Face free tier cold starts | First quiz generation takes 20–30s if model is unloaded | Upgrade to HF Pro or switch to Groq free tier |
| AI sometimes returns string correctAnswer | Fixed with Number() coercion but root cause is model inconsistency | Fine-tune a model or use a stricter structured output API |
| localStorage cleared by browser | History is backed by Supabase so this only affects the current session cache | Already mitigated — Supabase is source of truth |
| Rate limit: 5 requests/min | Users can't generate more than 5 quizzes per minute | Increase limit or add user-specific limits |
| No quiz sharing via unique URL | Share copies text, not a live URL | Implement quiz storage in Supabase and shareable quiz IDs |
| PDF export is basic | Plain jsPDF layout, no images | Add html2canvas for richer PDF export |
| No offline support | PWA manifest exists but no service worker | Add next-pwa package |
| Recharts not SSR-safe | Charts only render client-side | Use `dynamic()` with `ssr: false` if needed |
| Google OAuth requires setup | Users need to configure Google Cloud Console | Pre-configure for production deployment |
| AI hallucination on niche topics | Model may generate incorrect questions on very specialised topics | Add a "report incorrect question" feature |
